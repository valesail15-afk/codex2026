# 项目功能与流程规范文档 (Project Specification)

## 1. 项目概述
本项目是一个足球套利系统，旨在通过实时监控不同博彩平台（如竞彩、皇冠）的赔率差异，发现并计算套利机会，帮助用户实现低风险收益。

## 2. 功能清单 (Function List)

| 功能模块 | 功能描述 | 优先级 | 负责人 |
| :--- | :--- | :--- | :--- |
| **用户管理** | 支持管理员创建、锁定用户，设置用户使用时长阈值。 | 高 | 管理员 |
| **数据爬取** | 实时或模拟从竞彩、皇冠等平台抓取比赛及赔率数据。 | 高 | 系统/爬虫模块 |
| **套利引擎** | 基于线性规划算法，扫描单场及二串一的套利机会。 | 高 | 算法引擎 |
| **实时监控** | 通过 WebSocket 实时推送新发现的套利机会。 | 中 | 前端/后端 |
| **套利计算器** | 提供交互式界面，根据用户输入的本金自动计算各项下注额。 | 高 | 前端 |
| **历史记录** | 记录用户的下注详情、预期利润及实际结果。 | 中 | 用户 |
| **系统设置** | 配置返水率、分成比例、爬虫开关等全局参数。 | 中 | 管理员/用户 |

## 3. 业务流程图 (Business Process Flow)

### 3.1 核心套利流程
1. **数据采集**: 爬虫模块定时运行 -> 抓取赔率 -> 存入数据库 (`matches`, `jingcai_odds`, `crown_odds`)。
2. **机会扫描**: 
   - 触发条件: 数据更新后或用户手动刷新。
   - 逻辑: `ArbitrageEngine` 遍历所有 `upcoming` 比赛 -> 计算利润率 -> 过滤利润率 > 0.5% 的机会。
   - 存储: 结果存入 `arbitrage_opportunities` 和 `parlay_opportunities`。
3. **实时推送**: 后端通过 WebSocket 向在线用户发送 `refresh` 信号。
4. **用户决策**: 用户在“主面板”查看机会 -> 点击“计算下注” -> 进入“计算器”。
5. **方案执行**: 用户根据计算器给出的方案在各平台下注 -> 在系统中记录下注。

### 3.2 异常处理流程
- **数据缺失**: 如果某场比赛缺少竞彩或皇冠赔率，套利引擎将跳过该场比赛。
- **赔率变动**: 如果用户在计算时赔率发生变化，计算器会实时更新利润率，若利润转负，系统将显示警告。
- **时长到期**: 
   - 用户每分钟通过心跳接口同步时长。
   - 若 `used_duration >= max_duration`，后端返回 `403 DURATION_EXCEEDED`。
   - 前端拦截并弹出“时长到期”模态框，强制用户停止操作。

## 4. 模块间交互关系 (Module Interactions)

- **Frontend <-> Backend**: 通过 RESTful API 进行数据交换，通过 WebSocket 接收实时更新。
- **Backend <-> Database**: 使用 `better-sqlite3` 进行持久化存储。
- **Backend <-> ArbitrageEngine**: 后端逻辑调用引擎进行复杂的线性规划求解。
- **Crawler <-> External APIs**: 爬虫模块负责与外部数据源交互。

## 5. 关键操作步骤及注意事项 (Key Operations & Precautions)

### 5.1 管理员操作
- **创建用户**: 必须设置合理的 `max_duration`。
- **系统配置**: 修改返水率会直接影响套利利润率的计算，需谨慎操作。

### 5.2 用户操作
- **计算器使用**: 
   - 确保选择正确的“整数控制模式”（竞彩模式或皇冠模式）。
   - 下注前务必核实平台实时赔率是否与系统显示一致。
- **时长管理**: 留意页面右上角的剩余时间提醒。

### 5.3 开发规范
- **数据库变更**: 必须同步更新 `src/server/db.ts` 中的初始化逻辑，并考虑数据迁移。
- **API 变更**: 必须在 `AuthProvider` 的 Axios 拦截器中处理可能的异常状态码。
- **安全规范**: 严禁在前端暴露敏感配置，所有权限校验必须在后端完成。
