# 足球套利系统 (Football Arbitrage System) 产品规范文档

## 1. 架构规范 (Architecture Specifications)

### 1.1 技术栈
- **前端**: React 18+, TypeScript, Vite, Ant Design (UI 组件库), Tailwind CSS (样式).
- **后端**: Node.js, Express (API 服务), Vite Middleware (开发环境集成).
- **数据库**: SQLite (使用 `better-sqlite3`), 存储比赛数据、赔率及套利机会。
- **核心算法**: 
    - **单场套利**: 基础赔率对比与对冲计算。
    - **二串一套利**: 基于 **线性规划 (Linear Programming)** 的通用算法，使用 `javascript-lp-solver` 库。

### 1.2 核心逻辑组件
- **ArbitrageEngine**: 核心计算引擎，负责解析赔率、构建 LP 模型、计算最优下注额及利润率。
- **Crawler**: 爬虫模块，负责从数据源抓取竞彩与皇冠的实时赔率。
- **Database**: 数据持久化层，管理比赛、赔率及历史套利记录。

### 1.3 数据流
1. 爬虫抓取数据 -> 存入 SQLite。
2. 后端定时任务调用 `ArbitrageEngine` 扫描套利机会。
3. 扫描结果存入 `arbitrage_opportunities` 和 `parlay_opportunities` 表。
4. 前端通过 API 获取机会列表并展示。
5. 用户点击“计算下注”，前端携带参数跳转至计算器页面，实时调用引擎接口展示下注方案。

## 2. 核心套利算法 (Core Arbitrage Algorithm)

### 2.1 单场套利算法
- **目标**: 在竞彩买入某项（如主胜），在皇冠买入相反项（如客胜或受让）进行对冲。
- **计算公式**: 
    - 设竞彩投入为 $C$，赔率为 $O_{jc}$，返水为 $R_{jc}$。
    - 设皇冠投入为 $H$，赔率为 $O_{cr}$，返水为 $R_{cr}$。
    - 通过调整 $H$ 使 $C \times (O_{jc} + R_{jc}) \approx H \times (O_{cr} + R_{cr})$，确保无论结果如何，总回报大于总投入。

### 2.2 二串一套利算法 (通用 LP 模型)
- **赛果离散化**: 将每场比赛按净胜球分为 9 种可能（-4 到 +4，-4 代表输 4 球及以上，+4 代表赢 4 球及以上）。
- **线性规划建模**:
    - **变量**: 竞彩下注额 $C$，皇冠第一场各选项下注额 $H1_i$，皇冠第二场各选项下注额 $H2_j$。
    - **目标函数**: 最小化总成本 $T = C + \sum H1_i + \sum H2_j$。
    - **约束条件**: 在所有 81 种赛果组合（$9 \times 9$）下，总回报 $\ge 1$。
    - **总回报计算**: $C \times (\text{竞彩结算系数}) + \sum H1_i \times (\text{皇冠1结算系数}) + \sum H2_j \times (\text{皇冠2结算系数})$。
- **整数控制**: 求解后根据 `base_type` 进行缩放，使 $C$ 或 $\max(H)$ 满足 10,000 的倍数要求。

## 3. 套利规则 (Arbitrage Rules)

### 3.1 利润要求
- **最低利润率**: 扫描到的套利机会利润率必须 $> 0.5\%$。
- **全覆盖**: 方案必须覆盖所有可能的比赛结果，严禁出现亏损死角。

### 3.2 比赛选择
- **时效性**: 仅扫描近期 15 场比赛，确保赔率具有参考价值。
- **数据完整性**: 必须同时具备竞彩赔率和皇冠赔率（含亚盘）的比赛才进入计算。

### 3.3 结算系数规则
- **标准盘**: 胜/平/负完全匹配为 1.0，否则为 0。
- **让球盘 (Handicap)**:
    - 如 `-0.5`: 赢 1 球及以上为 1.0，否则为 0。
    - 如 `+0.5`: 赢球或平局为 1.0，输球为 0。
- **亚盘 (Asian Handicap)**:
    - 如 `-0.25`: 赢球为 1.0，平局为 0.5 (赢半)，输球为 0。
    - 如 `+0.25`: 赢球为 1.0，平局为 1.5 (收回本金+赢半)，输球为 0。

## 4. 交互流程 (Interaction Flows)

### 2.1 主面板 (Dashboard)
- **实时扫描**: 页面自动轮询或手动刷新获取最新套利机会。
- **详情查看**: 点击二串一机会，弹出 Modal 展示详细的投注组合（竞彩二串一 + 多个皇冠单场对冲项）。
- **跳转计算**: 点击“计算下注”，系统自动携带 `side` (竞彩方向) 和 `base_type` (整数控制) 参数跳转。

### 2.2 计算器 (Calculator)
- **自动初始化**: 从 URL 参数读取初始设置，无需用户手动输入。
- **自动计算**: 页面加载后立即执行计算逻辑，展示结果。
- **整数控制逻辑**:
    - **竞彩模式**: 确保竞彩下注额 $C$ 为 10,000 的倍数。
    - **皇冠模式**: 确保皇冠所有下注项中最大的金额为 10,000 的倍数。
- **结果展示**: 实时显示各赛果下的预期利润、利润率及总投入。

### 2.3 数据管理
- **手动刷新**: 提供全局刷新按钮，强制触发爬虫更新数据。

## 5. 设计规范 (Design Standards)

### 3.1 视觉风格
- **主色调**: 
    - 竞彩/品牌色: `#1890ff` (科技蓝)
    - 皇冠/警示色: `#fa8c16` (活力橙)
    - 利润/成功色: `#52c41a` (安全绿)
- **背景**: 浅灰色背景 (`#f0f2f5`) 配合白色卡片，保持界面整洁。

### 3.2 UI 组件 (Ant Design)
- **Card**: 用于封装独立的比赛信息或投注方案。
- **Table**: 展示套利机会列表，支持排序与筛选。
- **Statistic**: 突出显示利润率、总投入等关键指标。
- **Tag**: 用于标记投注方向（主胜、平局、客胜）及利润状态。

### 3.3 响应式与布局
- **栅格系统**: 使用 Ant Design 的 `Row` 和 `Col` 确保在不同分辨率下的兼容性。
- **间距**: 统一使用 Tailwind 的间距类（如 `p-4`, `m-2`, `gap-4`）。

### 3.4 字体与排版
- **字体**: 优先使用系统默认字体，备选 `Inter` 或 `PingFang SC`。
- **字号**: 正文 14px，标题 16px-20px，关键数字（利润）加粗并放大。

## 7. 项目蓝图 (Project Blueprint)

### 7.1 近期目标 (Short-term)
- **多平台接入**: 引入更多主流博彩平台（如 SBOBET, Pinnacle）的实时赔率。
- **自动化交易**: 实现与主流平台的 API 对接，支持一键下单或全自动对冲。
- **移动端适配**: 推出响应式移动端 H5 页面或小程序，方便用户随时随地监控机会。

### 7.2 中期目标 (Mid-term)
- **AI 赔率预测**: 利用机器学习模型分析历史赔率变动趋势，预判套利机会的出现。
- **社区化功能**: 建立套利者社区，分享策略、经验，并引入跟单系统。
- **多币种支持**: 支持主流法币及加密货币的结算与对冲。

### 7.3 长期愿景 (Long-term)
- **全自动套利生态**: 打造一个集数据采集、策略分析、自动执行、风险控制于一体的全球化套利生态系统。
- **合规化运营**: 在全球范围内寻求合规牌照，为用户提供安全、合法的套利服务。
- **资产管理**: 转型为专业的体育套利资产管理平台，为机构投资者提供高收益、低风险的投资标的。

## 8. 开发守则
- **类型安全**: 所有数据结构必须在 `src/types.ts` 中定义，严禁使用 `any`（除非 LP 库限制）。
- **错误处理**: 所有的 API 调用和计算逻辑必须包含 `try-catch`，并向用户展示友好的错误提示。
- **性能**: 二串一扫描应限制在近期活跃比赛内（如前 15 场），避免 LP 求解器过载。
